# Y2 Hetero-Precision Per-Layer Routing — Scoping Sketch (2026-05-19)

Y2 direction locked 2026-05-19 (project memory). Y1 paper §9 paragraph
positions it as "compiler-scheduled per-layer precision policy" on the
same Saturn FU lanes. This doc fills in the formulation, the DSL
extension shape, the May-Jul 2027 gate experiment, and the open questions
that the Aug 2026 scoping pass needs to close before any RTL/Exo work
restarts. Companion to `y2_firesim_prep.md` (validation infra) and
`precision_config.md` (Y1 locked recipe).

## 1. Problem statement

Y1 picks a **single per-stage precision recipe** (BF16 compute / NVFP4
K-V / FP32 softmax accum / FP8 attn weights / BF16 output) and applies
it uniformly across every transformer block of Llama-3.2-1B. The recipe
is hand-locked in `precision_config.md` and baked into a single
high-level `fa_kernel_decode_naive` @proc that the Exo precision-routing
pass lowers to a SaturnRVV kernel (`paper/exo_schedule_fa.py:5.5` in the
paper, 770 LOC).

Y2 question: **does a per-layer choice of precision recipe outperform
the uniform recipe at iso-accuracy** when the choice is generated by a
compiler scheduling pass and run on the same Saturn FU lanes?

Two layers can differ in:
- K-cache precision (NVFP4 vs BF16 — heavier cache layers may tolerate
  more aggressive compression than the first/last few that are known
  to be more outlier-sensitive in the PTQ literature).
- V-cache precision (same axis, independent of K).
- Attn-weight precision (FP8-E4M3 with per-row scale vs BF16, where
  the FP8 row-scale machinery is the J1 finding from
  `precision_config.md:24`).
- Softmax accumulator precision (FP32 always in Y1; per-layer relaxation
  to BF16 accum is *probably* unsafe — keep FP32 unless an experiment
  says otherwise).
- Output/residual precision (BF16 in Y1 — unlikely to vary).

Three of these five axes are non-trivial; the other two are likely
fixed. Per-layer config space is therefore ~2³ = 8 distinct configs
per layer × 16 layers ≈ 8¹⁶ ≈ 2.8×10¹⁴ unrestricted (1B; 3B = 28
layers ≈ 8²⁸ ≈ 2.0×10²⁵), but realistic search reduces this to a
small handful of monotone "first-N-layers high-precision, rest
low-precision" or "skip-band" patterns.

## 3. Formulation

### 3.1 Per-layer config

A per-layer config is a tuple
`(k_dtype_l, v_dtype_l, p_dtype_l)` for layer `l ∈ {0..L-1}`, drawn
from a small finite set (e.g. {NVFP4, BF16} × {NVFP4, BF16} ×
{FP8, BF16}). Y1's uniform recipe is the corner case
`(NVFP4, NVFP4, FP8) ∀ l`.

The full per-layer policy is a function
`π : layer_id → (k_dtype, v_dtype, p_dtype)`.

### 3.2 Objective

For a fixed Llama-3.2-1B and a fixed eval set (WikiText-2 perplexity +
C4 perplexity at a minimum; lm-eval-harness subset for downstream
accuracy if time permits), pick π to:

```
minimize    decode_latency(π)      # cycles per token on Saturn FU lanes
subject to  Δperplexity(π) ≤ ε     # iso-accuracy
            π(l) ∈ Allowed(l)      # feasibility (see §3.3)
```

Y1's uniform recipe gives a baseline `(latency_uniform,
Δppl_uniform)`. The Y2 gate is `latency(π*) ≤ 0.90 ×
latency_uniform` at `Δppl(π*) ≤ Δppl_uniform`.

### 3.3 Allowed set

Not every layer should be free to pick any precision. Three structural
constraints:

- **Monotonic K/V**: K-cache and V-cache for the same layer should
  use the same precision (KV pairs are read together; mixing them
  doesn't save bandwidth and complicates the dequant path).
- **First / last band**: prior PTQ work (ARCQuant arXiv:2601.07475
  among others) reports first 2-4 layers and last 1-2 layers are
  outlier-sensitive — force these to BF16 K/V by default. This shrinks
  the search space dramatically.
- **Smoothness**: π should not flip dtype every layer. Adjacent layers
  use the same K/V dtype unless an explicit "boundary" is justified.
  Encodes the prior that nearby layers behave similarly.

With these, per-layer policy reduces to a small parameter triple:
`(low_band_start, high_band_end, p_quant_layers)` — at most O(L²)
configurations, tractable for grid search.

## 4. DSL / Exo angle

The Y1 paper §9 paragraph (paper_draft.md:1397-1408) is explicit: the
novelty is the **scheduling-DSL machinery**, not a new low-precision
algorithm. Concretely:

### 4.1 What changes in Exo

The Y1 `exo_schedule_fa.py` pass exposes the precision recipe as
**Python-level constants** (`K_DTYPE`, `V_DTYPE`, `P_DTYPE`) that the
schedule reads when picking which `@instr` to inline at each
`replace()` site. Y2 lifts these to **schedule arguments** — the same
high-level `@proc` gets compiled N times (or once with a layer-id
switch), once per layer of Llama-3.2-1B.

Sketch:

```python
def lower_layer(precision: PerLayerConfig) -> ExoProc:
    p = fa_kernel_decode_naive
    if precision.k_dtype == NVFP4:
        p = replace(p, "_ = K[...]", saturn_vfconv_nvfp4_bf16)
    elif precision.k_dtype == BF16:
        p = replace(p, "_ = K[...]", saturn_vle_bf16)
    # ... same for V, P ...
    return simplify(p)

kernels = [lower_layer(policy[l]) for l in range(L)]
```

The Y1 pass becomes a degenerate case where `lower_layer` is called
with the same config for every l.

### 4.2 What does NOT change

- The 4 Saturn customs. The lanes are the same; the per-layer policy
  just picks subsets.
- The §6 fused-FA kernel shape (per-row dot + softmax + per-key
  scalar-broadcast). Independent of the precision choice.
- The gem5 + FireSim eval harness.

### 4.3 Why "compiler-DSL framing"

If Y2 reframed as "we picked per-layer NVFP4 vs BF16 by perplexity
search," ARCQuant already covers that algorithmically — it's an
*algorithm contribution*, not an *architecture* one. The compiler-DSL
framing makes the contribution: "the scheduling-DSL machinery that
takes a per-layer policy and emits N specialised kernels onto the
shared Saturn lanes is the artifact." The policy itself is an input;
the search-to-emission pipeline is the output. This matches HPCA /
MICRO's appetite for HW-compiler co-design over pure-algorithm work.

## 5. Search machinery

### 5.1 Compile-time grid search (recommended)

For each candidate π in the Allowed set (§3.3), compile the N
specialised kernels via Exo, run a calibration pass on a small
held-out set (say, WikiText-2 first 256 sequences), record both
`Δppl(π)` and a *cycle-count proxy* (gem5 single-layer decode time
× L). Pick the Pareto front; report the knee config.

Estimated cost: |Allowed| ≈ 30-100 configs × gem5 runtime per config
(~5-10 min for a single decode step) × calibration set size → a few
GPU-days at most on the existing i9-13900 simulator host. No paid
hardware needed.

### 5.2 Why not runtime policy

A runtime policy (pick precision per token / per request) is a more
ambitious framing but: (a) collides with the "FU lanes are static
hardware" framing, (b) overlaps with adaptive-precision algorithm
work, (c) is harder to validate. Defer to Y3 or later.

### 5.3 Why not RL / Bayes-opt

Search space is tractable for grid; reaching for RL is over-engineering
for the contribution. The interesting machinery is the **compiler
lowering**, not the search algorithm.

## 6. May-Jul 2027 gate experiment

Single decision point, **timed at May-Jul 2027** — chosen because: (a)
MLArchSys'27 submission lands Apr 2027, reviews come back ~May 2027,
so the gate fires while MLArchSys'27 author-response work is still
active and Y2 commitment can fold in any reviewer signal from the Y1
artifact; (b) MICRO'28 deadline is ~Apr 2028, so a May-Jul 2027 gate
leaves ~9 months for full Y2 execution if the gate passes — tight but
feasible. HPCA'28 (Aug-Sep 2027 abstract / Jan 2028 paper) is **reach
only if the gate fires early (~May 2027) and Phase A from Q7 lands
clean Nov 2026**; under any slip it falls to MICRO'28. **MICRO'28 is
the primary venue; HPCA'28 is reach.** The phrase "Q1 2027 gate" in
earlier project memory was sloppy wording — the actual constraint is
"post-MLArchSys'27, pre-MICRO'28 prep window."

**Gate question: does per-layer routing beat uniform routing by ≥10 %
decode latency at iso-accuracy on Llama-3.2-1B?**

### 6.1 Setup

- Model: Llama-3.2-1B (16 layers, n_attn_heads=32, n_kv_heads=8
  GQA, head_dim=64, hidden=2048; verified 2026-05-19 against HF
  config.json). Download to `models/` (not yet present; `models/`
  currently has qwen2.5-0.5b only). Llama-3.2-3B for camera-ready
  ablation: 28 layers, n_attn_heads=24, n_kv_heads=8 GQA,
  head_dim=128, hidden=3072 — note 3B's head_dim is **double** 1B's,
  so 3B KV-cache size scales as 28/16 × 128/64 = 3.5× per-token vs 1B.
- Eval: WikiText-2 ppl + C4 ppl (mandatory); lm-eval-harness 5-task
  subset (nice-to-have, ARC-easy / HellaSwag / WinoGrande / PIQA /
  BoolQ).
- HW: gem5 RiscvO3CPU + DDR3-1600 (same as Y1 paper §7). FireSim
  validation only after gate passes.
- Iso-accuracy tolerance: `ε = Δppl_uniform + 0.1 ppl` (i.e. Y2
  policy must not lose more than 0.1 ppl on top of Y1's uniform
  recipe's loss).

### 6.2 Decision tree

| Result | Action |
|--------|--------|
| ≥10 % speedup, ≤ε ppl loss | **Commit Y2**. Begin full HPCA'28 / MICRO'28 paper drafting; bring FireSim validation online via `y2_firesim_prep.md`. |
| 5-10 % speedup, ≤ε ppl loss | **Tighten**. Try (a) Llama-3.2-3B (bigger model = more layers = more flexibility), (b) larger seq_len (cache-bound regime where NVFP4 wins more), (c) add P-quant variation that Y1 holds fixed. Re-test at end of Q2 2027. |
| <5 % speedup or >ε ppl loss | **Fall back to FireSim validation paper**. Y1 → FireSim validation (`y2_firesim_prep.md`) becomes the standalone Y2 contribution (methodology paper at MLArchSys'28). Y3 (spec decode on Saturn) remains on track. |

### 6.3 What this gate avoids

The gate exists to *kill* Y2 before the 12-month investment if the
per-layer idea doesn't have legs at small-model scale. Without it,
Y2 risks delivering a negative result close to the HPCA / MICRO
deadlines with no fallback.

## 7. Differentiation from concurrent work

### 7.1 ARCQuant (arXiv:2601.07475)

Most-cited overlap. ARCQuant is an **algorithm** contribution: it picks
which weights to keep in higher precision for NVFP4 PTQ. Y2 is a
**compiler/architecture** contribution: it picks which *kernels* to
specialise to which precision across layers, on a fixed HW substrate.

The differentiation must be in writing from the abstract on. Compiler-
DSL machinery, Exo schedule attribute, Saturn FU integration: those are
the words that signal the framing. "Per-layer precision policy"
without further qualification reads ARCQuant-adjacent and risks scoop
framing.

Mitigation: in §1 of Y2 paper, explicit one-paragraph diff vs
ARCQuant. Cite their PPL numbers as our quant-algorithm baseline.

### 7.2 VMXDOTP (arXiv:2603.04979)

Same axis as Y1's diff: RVV MX (not NVFP4), unfused GEMM (not fused
FA), in-order Spatz (not Saturn OoO), hand-tuned (not Exo), DeiT image
(not LLM E2E). Y2's per-layer angle is **further** orthogonal —
VMXDOTP is single-precision per workload. Should be a half-sentence
to dispatch.

### 7.3 Han lab (sparse/streaming attention)

Adjacency risk: if Song Han's group lands a per-layer sparsity policy
that uses similar compiler-scheduled framing, our story rhymes too
closely. Mitigation:
- Email Han lab before Y2 paper-draft circulation. The Y1 paper is
  already public; positioning Y2 ahead of any Han-lab equivalent is
  the play.
- Frame Y2 strictly on the **precision** axis, not the **sparsity**
  axis. They're orthogonal contributions and can coexist.

### 7.4 Ragan-Kelley group (Exo upstream)

Y2 extends our Exo platform fork (`exo-saturn-rvv`). If Ragan-Kelley
upstream lands a per-target precision-policy primitive in Exo trunk
between now and May-Jul 2027, we lose part of the "novel DSL machinery"
framing. Mitigation: email-before-publish, possible co-authorship
discussion if they want to merge our work upstream.

## 9. Aug 2026 open questions

When Aug 2026 pickup starts (warmup_prompt's Aug-Oct window), these
need answers before any code lands:

1. **Llama-3.2-1B vs 3B for gate? — RECOMMENDATION: 1B for gate,
   3B for camera-ready ablation.**

   *Why 1B for the gate:* (a) iteration speed dominates a kill
   decision — sweeping the ~30-100 candidate π's of §5.1 fits in
   a few days on 1B vs a week+ on 3B at gem5 RiscvO3CPU rates;
   (b) Y1 paper §7 + `precision_config.md:62` already commit to
   Llama-3.2-1B, so 1B per-layer numbers diff directly against
   Y1's uniform-policy baseline with no model-change confounder;
   (c) the gate is an internal kill decision, not a reviewer
   artifact — reviewer-credibility concerns belong to camera-ready.

   *Why not 3B for the gate:* layer-count headroom is marginal
   (28 vs 24 ≈ 17 %, not transformative — per-layer payoff depends
   mostly on outlier-band size + KV-BW share, similar at both
   scales). Slower per-config eval steals from the search budget
   and yields a noisier gate signal.

   *Where 3B earns its keep:* (a) if 1B gate passes, run the same
   policy family on 3B as camera-ready generalisation evidence;
   (b) if 1B gate is borderline (5-10 %, "tighten" branch in §6.2),
   3B is the first tightening move — bigger model often = bigger
   NVFP4 BW win.

   *Layer-count correction (verified 2026-05-19):* the
   previously-cached "24 layers" claim in project memory +
   `precision_config.md:62` was wrong. HuggingFace `config.json`
   for `meta-llama/Llama-3.2-1B` confirms **16 hidden layers**
   (also: 32 attn heads, 8 KV heads, head_dim=64, hidden=2048).
   Llama-3.2-3B has **28 hidden layers**, head_dim=**128**,
   hidden=3072 — note 3B's head_dim is double 1B's, so 3B per-
   token KV cache is ~3.5× larger than 1B's. This sharpens the
   "3B for camera-ready" case: gate-vs-camera-ready layer-count
   ratio is 28/16 = 1.75× (more layer-policy headroom than the
   earlier 28/24 = 1.17× estimate), and 3B's larger per-token KV
   cache amplifies the NVFP4 BW win the policy can chase. Gate
   recommendation (1B) is unchanged. `paper_draft.md:248-250`
   and `precision_config.md:62` corrected 2026-05-19.

   *Second-order caveat:* if reviewer norms by Q3 2027 have shifted
   toward 7B+ baselines for precision-policy work, add a 7B
   single-policy ablation alongside 3B. 7B on gem5 is expensive
   but tractable for one policy run, not a full sweep.
2. **First / last band size? — RECOMMENDATION: start with band =
   (2 first, 1 last) on 1B; grow only if empirical Δppl forces it.**

   *Why start at (2, 1):* (a) PTQ literature (LLM.int8 Dettmers
   NeurIPS'22, AWQ Lin MLSys'24, OmniQuant ICLR'24) consistently
   identifies first 2-4 + last 1-2 layers as the outlier-sensitive
   band — pick the *smaller* end of each range because Y2 wants
   maximum per-layer freedom, not maximum safety; (b) with 1B's
   16 layers, even (2, 1) consumes 19 % of layers and leaves 13
   free to vary — enough room for the per-layer policy to show
   payoff; (c) (3, 2) consumes 31 % (only 11 free layers), which
   may cap policy payoff before the gate experiment can detect it.

   *Why not trust literature blindly:* the (2-4, 1-2) heuristic
   was developed for INT8 / INT4 **weight** quant. KV-cache NVFP4
   outlier behavior may differ — KV is activation-derived, not
   weight-derived, and per-layer KV-magnitude profiles on
   Llama-3.2 have not been (publicly) measured for NVFP4
   specifically. ARCQuant's <0.5 ppl on Llama-3.2 NVFP4 K/V
   likely relies on per-channel calibration rather than explicit
   band protection — so the band may turn out to be unnecessary
   entirely, or to live at unexpected layer indices.

   *Empirical verification (Aug 2026 first task after gate setup):*
   uniform-NVFP4-K/V-except-band sweep on first 100 WikiText-2
   sequences. Sweep band = {(0, 0), (1, 0), (2, 1), (3, 1),
   (4, 2), (full-NVFP4)}. Smallest band that keeps Δppl ≤ ε is
   the locked starting point for the per-layer search. Total
   cost: 6 calibration runs × ~10 min gem5 fwd-pass each ≈ 1 hr
   on the existing simulator host. Cheap.

   *Caveat for 3B camera-ready:* with 28 layers, (2, 1) is 11 %
   of layers and (4, 2) is 21 %. 3B has more headroom for a
   larger band without choking the policy — re-decide the band
   for 3B independently of 1B's gate-locked band.

   *Open sub-question deferred to empirical run:* if the per-layer
   sensitivity pattern is **not** monotone (e.g. outliers
   concentrate in the middle layers, as some streaming-attention
   work suggests for long-context inference), the "band"
   parameterisation is wrong and §3.3's Allowed-set constraint
   needs to switch to "select any K of L layers for high
   precision." This re-opens the search space from O(L²) to
   C(L, K), still tractable for small K but worth flagging now.
3. **Compile-time vs runtime policy framing? — RECOMMENDATION:
   compile-time, with a "per-context-length bucket" relaxation
   allowed if Aug-2026 sensitivity sweep demands it. Runtime
   dispatch explicitly deferred to Y3 / follow-up.**

   *Why compile-time:* (a) **contribution clarity** — compile-time
   per-layer policy frames Y2 as "scheduling-DSL machinery emits
   N specialised kernels onto Saturn lanes," a clean
   HW-compiler co-design story; runtime smears the contribution
   across compiler + dispatch logic + (implicit) decision policy
   and starts to look like an algorithm paper with HW
   appendix; (b) **differentiation from ARCQuant** — compile-time
   reads as "policy is a schedule attribute" (architecture/
   compiler novelty); runtime reads as "policy is an inference
   decision" (algorithm novelty), which is ARCQuant's home turf;
   (c) **evaluation tractability** — compile-time has a single
   deterministic per-layer cost; the gate experiment is
   well-defined and clears or fails cleanly. Runtime has per-input
   variance, needs distribution-level error bars, and risks a
   noisy gate signal that hides true policy payoff; (d) **artifact
   size fits 12 months** — compile-time is ~500 LOC Exo-pass
   extension + N emitted kernels; runtime adds a dispatch table,
   monitor, decision logic, and llama.cpp harness changes — easily
   2-3× the engineering scope. Y2 already includes Llama-3.2-1B
   integration + FireSim Phase A-C; runtime would push the
   schedule beyond the Apr 2028 HPCA / MICRO window.

   *Why not runtime:* the framing collides with adaptive-precision
   algorithm work (`KVQuant`, ARCQuant, streaming-attention),
   where the contribution is the decision policy itself. Y2's
   HW + compiler novelty is sharpest when policy is *static*
   (you compile it in) and the interesting machinery is
   "compiler lowering N policies → N kernels." Adding runtime
   dispatch makes the boundary fuzzy and invites reviewer
   pushback ("isn't this just dynamic quantization with an FU?").

   *Allowed relaxation — per-context-length buckets:* if the Aug
   2026 sensitivity sweep shows that optimal policy depends
   meaningfully on context length (e.g. short-context favors
   higher precision because KV is small; long-context favors
   aggressive NVFP4 because KV dominates BW), pre-compile separate
   policies per context-length bucket (e.g. 0-2K, 2K-8K, 8K+) and
   dispatch by context-length at call entry. This is still
   "compile-time" in spirit (no runtime decision logic; just a
   table lookup on a single integer) and captures the dominant
   axis of workload variation. Adds ~3× kernel count but
   no real artifact-size increase.

   *Caveat — Han-lab moat:* Han lab's streaming-attention work
   may publish a runtime per-layer/per-token precision policy as
   a competing Y2-window contribution. Compile-time framing is
   the cleanest defensive moat — runtime per-layer is contested
   territory and we'd lose framing if scooped. This argues for
   compile-time *even* if a runtime variant turns out
   technically better; the framing battle matters more than
   marginal perf for a Y2 publication.

   *Frame runtime as future work in Y2 §9:* one paragraph stating
   "runtime per-token / per-request precision adaptation is the
   natural follow-on; we keep policy compile-time to isolate the
   HW + compiler contribution." Preserves the option for Y3 or
   a separate paper without committing to it now.
4. **Calibration set? — RECOMMENDATION: WikiText-2 + C4 perplexity
   only for the gate. Add lm-eval-harness 5-task subset (ARC-easy,
   HellaSwag, WinoGrande, PIQA, BoolQ) for camera-ready.**

   *Why ppl-only for the gate:* (a) **iteration speed** — gate
   sweeps ~30-100 candidate π's × decode-trace eval; ppl on first
   1000 WikiText-2 + 1000 C4 sequences runs in ~5 min per config
   on the i9-13900 host (no GPU needed), so the full sweep
   completes overnight. lm-eval-harness 5-task adds ~30 min per
   config → 25 hr of additional eval per sweep, prohibitive at
   the iteration cadence the gate requires; (b) **kill-decision
   precision adequacy** — gate is a 10 % speedup ratio decision,
   not a precision absolute. Δppl correlates strongly enough
   with downstream task accuracy at the *gross* deltas (>0.5 ppl)
   that distinguish gate-pass from gate-fail; the fine-grained
   downstream picture only matters at camera-ready when reviewers
   ask "did your 10 % speedup cost real accuracy"; (c) **standard
   PTQ-paper baseline** — both ARCQuant and the GPTQ / AWQ /
   OmniQuant literature report WikiText-2 + C4 as the primary
   numeric, so the gate signal is directly comparable to those
   baselines without methodology gymnastics.

   *Why add 5 lm-eval-harness tasks for camera-ready:* (a)
   reviewer norms have shifted post-2024 — ppl-only is rejection
   bait at HPCA/MICRO; expect at least one reviewer to ask for
   task accuracy; (b) the 5-task subset (ARC-easy, HellaSwag,
   WinoGrande, PIQA, BoolQ) is the *de facto* standard quick-eval
   bundle used by ARCQuant et al., fast enough to fit camera-ready
   timeline (~5 hr for the locked policy + uniform baseline +
   FP16 reference = 3 configs × 30 min ≈ 1.5 hr), and gives task-
   accuracy delta numbers that match the dominant Y2 framing.

   *Specific gate-stage protocol:*
   ```
   For each π in Allowed:
     calibrate per-channel KV scales on WikiText-2 [0:256]
     measure Δppl on WikiText-2 [256:1256] + C4 [0:1000]
     measure decode latency on gem5 (1 layer × Llama config × seq_len=2048)
   Pareto front by (decode_latency, Δppl)
   ```

   *Iso-accuracy threshold:* keep `ε = Δppl_uniform + 0.1 ppl` as
   stated in §6.1. The 0.1 ppl margin accounts for calibration
   noise across the ~30-100 configs; tightening to `+0.05` risks
   killing viable policies on stochastic ppl fluctuation.

   *Caveat:* if the gate signal is borderline (5-10 % speedup
   band per §6.2 "tighten" branch), re-run with lm-eval-harness
   on the top-5 Pareto policies before declaring gate-fail. Cheap
   safety net (~5 configs × 30 min ≈ 2.5 hr added) that may
   rescue a marginal pass.
5. **Should FP8 attn-weight precision be in the search space? —
   RECOMMENDATION: PIN to FP8 (matches Y1) for the gate
   experiment. Open as a search axis only if the K/V-only gate
   shows marginal payoff (5-10 % band) and you want to expand
   the search before falling back.**

   *Why pin for the gate:* (a) **isolate the variable** — the
   gate's job is to answer "does per-layer policy beat uniform?"
   on the *cheapest, highest-leverage* axis. KV precision is the
   dominant BW axis (3.56× compression, every decode token reads
   the whole KV cache); attn-weight precision is a transient
   intermediate (FP8 vs BF16 = 2× on a per-row buffer, no
   persistence). Opening both axes at gate-time confounds the
   signal — a 12 % speedup could come mostly from attn-weight
   policy, not from per-layer KV routing, and the contribution
   framing would slip; (b) **engineering cost** — the J1
   per-row FP8 scale machinery (`precision_config.md:24-49`)
   adds ~30 LOC per kernel variant for the bf16-quant case
   (extra scalar FMUL per output dim, row-scale state, per-row
   max). Per-layer toggling means the dispatch table holds 2×
   the kernel variants and the BF16-vs-FP8 boundary is in the
   fused kernel's hot loop — non-trivial Exo pass extension;
   (c) **ARCQuant overlap risk grows** — per-layer attn-weight
   precision is closer to ARCQuant's algorithm-paper territory
   (they pick which weights stay high-precision; we'd be picking
   which layers' attn-weight buffers stay BF16). Compiler-DSL
   framing weakens if the search space looks like a
   quantization algorithm.

   *Why open the axis at "tighten" branch (§6.2 5-10 % band):*
   if K/V-only per-layer gives ~7 % speedup and we need ≥10 %,
   adding the attn-weight axis adds search dimensions that
   *might* push past the threshold. Worth trying before falling
   back to FireSim-validation paper.

   *Three things to keep pinned even at "tighten" time:* (i)
   FP32 softmax accumulator — going below FP32 is reviewer-
   rejection bait per `precision_config.md:53-58`; (ii) BF16
   output — varying this affects residual stream, breaks
   downstream layer accuracy; (iii) FP8 row-scale machinery
   itself — the J1 finding is necessary for **any** FP8 attn-
   weight policy at long context, not optional.

   *Recommendation summary table:*
   | Axis | Gate (1B) | Tighten (1B if 5-10 %) | Camera-ready ablation (3B) |
   |------|-----------|------------------------|----------------------------|
   | K-cache dtype | varies per layer | varies per layer | varies per layer |
   | V-cache dtype | tied to K | tied to K (or open) | tied to K |
   | attn-weight dtype | **pinned FP8** | open (if K/V-only marginal) | open |
   | softmax accum | pinned FP32 | pinned FP32 | pinned FP32 |
   | output dtype | pinned BF16 | pinned BF16 | pinned BF16 |
