1. **Llama-3.2-1B decode tok/s** — ours (Saturn + 4 custom FUs + mixed-prec) vs upstream generic RVV FA vs SpacemiT K1 VLEN=1024 FA (FP16) vs ARM Cortex-X4 commodity reference.
2. **KV-cache memory bandwidth (GB/s, decode)** — ours (NVFP4 K/V) vs FP16 K/V baseline. **Mo 2 headline number.**
3. **Energy per token (mJ)** at iso-throughput — ours vs SpacemiT K1.
4. **Prefill latency (1k tokens)** — same four platforms.
5. **Precision-routing FU primitives** — cycle latency + area for each of `vfconv.nvfp4.bf16.v`, `vfconv.bf16.fp8.v`, `vfconv.fp8.bf16.v`, `vfexp.v` (4 numbers each: cycle, area mm², area % Saturn, max error vs scalar reference).
6. **PPA delta**: Saturn + precision-routing FU vs unmodified Saturn @ 16 nm est. via Yosys/OpenROAD.
7. **Exo-generated kernel vs hand-coded** RVV intrinsics — parity check on cycle count, correctness vs FP16 reference.
8. **Llama-3.2-1B perplexity** (WikiText-2 + C4) — mixed-precision (BF16 logits / NVFP4 K-V / FP32 softmax / FP8 attn weights / BF16 output) vs FP16 baseline. **Target: <1.0 perplexity points degradation.**

## Mo 2 measurement detail (highest immediate priority)

**Question**: Does NVFP4 K/V cache reduce decode memory bandwidth by ≥30% vs FP16 K/V on the Saturn-targeted RVV path?

**Approach** (zero-cost, gem5 SE-based, no silicon needed):
1. Write a small NVFP4 dequant + matmul microbench in C with RVV intrinsics targeting Saturn-like dimensions:
   - KV-cache representative shape: GQA n_kv_heads=8, head_dim=64, seq_len=2048
   - One iteration: dequant K (NVFP4 → BF16) + GEMV (Q × K^T)
2. Compare against the FP16-K-cache GEMV (using existing `__riscv_vle16` + `__riscv_vfmul`)
3. Run both under gem5 SE with `--cpu-type=TimingSimpleCPU` + simple cache hierarchy (32 KB L1D, 512 KB L2)
4. Capture from gem5 stats:
   - `system.cpu.dcache.overallAccesses::total` (cache accesses)
   - `system.cpu.dcache.overallMisses::total` (misses → DRAM bytes)
   - `system.mem_ctrls.bytesRead::total` (DRAM bytes read)
   - cycle count
5. Compute bandwidth ratio: `(FP16 bytes / FP16 cycles) / (NVFP4 bytes / NVFP4 cycles)`

**Analytical anchor**: NVFP4 K/V with block-16 scale = 4.5 bits/element vs FP16 = 16 bits/element → 3.56× compression. Saturn-bound decode at seq_len=2048 should see ~3× actual BW reduction (some overhead from dequant lane). Mo 2 threshold (≥30%) comfortably exceeded analytically; gem5 confirms.

**Reusable infrastructure for Mo 2**:
- Cross-compile: `riscv64-linux-gcc -march=rv64gcv -mabi=lp64d -static -O2 microbench.c -o microbench`
- gem5: `gem5.opt configs/deprecated/example/se.py --cpu-type=TimingSimpleCPU --num-cpus=4 -c microbench`
- All paths already verified on this host (see `handoff.txt`)

**If Mo 2 fails (<30% BW reduction)**:
- First diagnose: is it the dequant lane overhead? Run NVFP4 K-cache GEMV WITHOUT dequant cost (pre-dequanted in memory) → upper bound on possible win.
- If upper bound is <30%, NVFP4 K/V isn't the right pitch — pivot back to options B/C/D from the widening menu.
- If upper bound is ≥50% but our actual is <30%, the dequant FU is the bottleneck → optimize the `vfconv.nvfp4.bf16.v` cycle latency before paper submission.

## Critical-path warnings

- **Verilator too slow for end-to-end Llama.** Microbench only on Verilator. FireSim by month 8.
- **ASIC synth = long pole.** OpenROAD/Yosys on a Chipyard SoC is 2-4 wk wall time after RTL stable. Start month 9.
- **Energy methodology consistency**: K1 borrow (if granted) = full-board power; FireSim = chip-only. Pick one frame for the abstract; footnote the delta.
- **NVFP4 calibration**: needs ARCQuant tooling OR hand-rolled NVFP4 quantizer on Llama-3.2-1B before perplexity numbers are real. Don't wait until Mo 10 to discover quant calibration matters.
- **Saturn FU integration risk**: the `FunctionalUnitFactory` pattern is from project memory; verify against current `saturn-vectors` source before Mo 3.
- **Exo `@instr` precedent**: Exo trunk added ARM SVE May 2025 but NO RVV yet. If Ragan-Kelley group publishes Exo-RVV before our Mo 8, the compiler-co-design lead weakens. Set arXiv keyword alerts; email Exo authors early as collaboration / heads-up.
- **VMXDOTP follow-up risk**: Benini group (VEXP, VMXDOTP) ships RVV-customs-for-ML papers regularly. Plausibly an attention-themed sequel for HPCA/MICRO 2027 timed close to our submission. Differentiate hard on (a) NVFP4 vs MX, (b) fused FA vs unfused GEMM, (c) Exo vs hand intrinsics, (d) Saturn vs Spatz, (e) Llama-3.2-1B vs DeiT.
