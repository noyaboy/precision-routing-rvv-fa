## Abstract (draft 1, ~225 words)

Edge LLM inference is shifting to RISC-V silicon — SiFive Intelligence, SpacemiT K1, Tenstorrent Ascalon, and Andes AX65 ship RVV 1.0 cores today — and the open RVV software stack has caught up: upstream llama.cpp now provides a polynomial RVV softmax kernel and a K1-tuned fused flash-attention path. The gap that remains is *precision routing*. Edge-LLM decode is memory-bandwidth-bound — every generated token reads the entire KV cache — and the open RVV stack pins all attention stages to one precision, leaving the headline KV-cache bandwidth savings (3.2× from 4-bit microscale formats like NVFP4) unrealized. Existing fused-attention hardware (FuseMax, MICRO 2024) likewise pins one precision; concurrent RVV custom-instruction work (VEXP ARITH 2025, VMXDOTP DATE 2026) accelerates individual ops but not the full attention pipeline with mixed precision. We present a co-designed mixed-precision fused flash-attention stack: (i) four new RVV custom instructions on the Berkeley Saturn vector unit — `vfconv.nvfp4.bf16.v`, `vfconv.bf16.fp8.v`, `vfconv.fp8.bf16.v`, and a pipelined `vfexp.v` — implementing the precision-conversion lanes attention needs (~0.14 mm² @ 16nm, ~3% of Saturn); (ii) an Exo 2 precision-routing scheduling pass that emits VLEN-parameterized fused flash-attention kernels routing per stage (BF16 logits / NVFP4 K-V cache / FP32 softmax accumulator / FP8-E4M3 attention weights); (iii) end-to-end integration in llama.cpp. On Chipyard FireSim and against SpacemiT K1's hand-tuned VLEN=1024 fused FA, we achieve [N]× Llama-3.2-1B decode speedup at <1.0 perplexity degradation. All RTL, compiler passes, and kernels open-source.

Keywords: RISC-V, RVV, large language models, flash attention, mixed precision, NVFP4, compiler co-design, scheduling DSL, open hardware.

## Hook paragraph (draft 1, ~210 words)

Edge LLM inference has shifted from research to production: Apple Intelligence, Llama-3.2-1B/3B, Phi-3.5 mini, and Gemini Nano run on consumer devices, and on-device serving is increasingly the default. RISC-V is the emerging open-hardware substrate — SiFive Intelligence X-series, SpacemiT K1, Tenstorrent Ascalon, and Andes AX65 ship RVV 1.0 cores today. The open software stack has matured along with the silicon: upstream llama.cpp now ships a polynomial RVV softmax kernel (`ggml_v_expf_m2`) and a SpacemiT-tuned VLEN=1024 fused flash-attention path. But edge-LLM decode is *memory-bandwidth-bound* — each generated token reads the entire KV cache — and the open RVV stack pins all attention stages to one precision, leaving the headline KV-cache bandwidth savings (3.2× from 4-bit microscale formats like NVFP4 [ARCQuant 2026]) unrealized. The fundamental gap is not a missing op (softmax is vectorized) or a missing fused kernel (SpacemiT shipped one for K1) — it is the *co-design loop* between custom precision-conversion hardware and a compiler that decides safely which precision per attention stage. Existing fused-attention HW (FuseMax, MICRO 2024) pins FP16 throughout; concurrent custom-instruction work targets individual ops (VEXP scalar softmax-exp; VMXDOTP unfused GEMM). **We close this loop with four new RVV instructions on the Berkeley Saturn vector unit, an Exo 2 precision-routing scheduling pass, and an end-to-end llama.cpp integration on Llama-3.2-1B. All RTL, compiler passes, and kernels are open-source.**

## Contributions (draft 1)

1. **Per-stage precision-routing analysis for fused attention.** We identify KV-cache memory bandwidth as the edge-LLM decode bottleneck and decompose attention into stages with different precision tolerance: logits (BF16), softmax sum-accumulator (FP32), KV cache (NVFP4 block 16 with E4M3 scale), attention weights (FP8-E4M3), output (BF16). We characterize the perplexity / area / bandwidth trade-off of each stage independently on Llama-3.2-1B. To our knowledge, this is the first per-stage precision-routing analysis for fused-attention on an open vector ISA; FuseMax pins one precision throughout and ARCQuant's NVFP4 work is algorithm-only.

2. **Four RVV custom instructions on Saturn.** We design and implement `vfconv.nvfp4.bf16.v` (3-cycle pipelined NVFP4 dequant with block-16 E4M3 scale broadcast), `vfconv.bf16.fp8.v` (2-cycle BF16→E4M3 quant with saturation), `vfconv.fp8.bf16.v` (1-cycle E4M3→BF16 dequant), and `vfexp.v` (8-cycle pipelined exp via range-reduce + degree-5 minimax + reconstruct). Implemented as new functional units via Saturn's `FunctionalUnitFactory`. Total area: ~0.14 mm² @ 16nm, ~3% of Saturn baseline. RTL open-source; reusable on Ara and T1 with mechanical port changes.

3. **Exo 2 precision-routing pass.** We extend the Exo 2 scheduling DSL [Exo 2025] with `@instr` declarations for the four custom instructions and a precision-routing pass that maps each attention stage to its assigned dtype, emitting a single fused flash-attention kernel covering QK^T → softmax → ·V. The pass is VLEN-parameterized: the same Exo specification generates correct kernels for VLEN=128/256/512. Prior Exo backends target AVX-512, NEON, Gemmini, and RVM; ours is the first RVV 1.0 backend with custom-instruction awareness. (Verify before submission: Ragan-Kelley group has not published Exo-RVV as of 2026-05-16 recheck; Exo trunk added ARM SVE May 2025, nothing RVV.)

4. **Fused flash-attention kernel on Saturn.** Our Exo-generated kernel uses the four custom instructions in a single pass: NVFP4 K/V dequant streamed into BF16 matmul, vfexp inside softmax with FP32 accumulator, FP8-E4M3 attention weights re-quantized for the second matmul. We demonstrate correctness against the FP16 reference on Llama-3.2-1B and bit-equivalence with a hand-coded RVV reference at fixed VLEN. Throughput targets: ≥0.9× hand-coded at VLEN=256 (Mo 8 checkpoint).

5. **End-to-end Llama-3.2-1B inference in llama.cpp.** We integrate the Exo-generated kernel into llama.cpp's RVV path via a new `GGML_RVV_NVFP4` backend variant and report Llama-3.2-1B decode throughput, energy per token, and prefill latency on Chipyard FireSim (Saturn + Rocket SoC) vs (a) upstream generic RVV FA, (b) SpacemiT K1 hand-tuned VLEN=1024 FA, (c) ARM Cortex-X4 commodity reference. Quantitative claim: ≥1.5× decode speedup at iso-perplexity (≤1.0 perplexity points degradation on WikiText-2 + C4) vs K1 (Mo 10 checkpoint).

6. **Open-source artifact.** All Saturn RTL fork, Chisel testbenches, Exo `@instr` library, scheduled kernels, llama.cpp patch series, benchmark scripts, and NVFP4 calibration data open-source under permissive license. Upstream PRs filed against Saturn (RTL), Exo (instruction library), and llama.cpp (NVFP4 backend variant). PR strategy: small upstream patches early (e.g., VLEN-parameterized generic fused FA generalizing SpacemiT's path, before NVFP4 lands).
